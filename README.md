# AI 面试题库问答助手

> 一个基于**真实面经**的 AI 面试问答助手。用户提问某个公司/岗位的面试问题,助手用 **RAG(检索增强生成)** 技术回答:既能溯源真题,又能给出参考答题思路。
>
> 面向秋招 **AI 产品经理 / AI 产品运营** 岗位求职者。从自身备考痛点出发,独立完成从需求定义、产品设计、技术开发到部署上线的全流程。

---

## 一、解决什么问题

| 痛点 | 本产品的解法 |
|---|---|
| 面经散落在牛客、小红书等平台,收集困难、真假难辨 | 手动收集真实面经,构建结构化知识库 |
| 直接问通用大模型,会**编造**高频题,不可信、无法溯源 | 用 RAG 让回答基于真实面经,可信可溯源 |
| 学生不只想知道"被问了什么",更想知道"该怎么答" | 分层回答:真题溯源 + AI 补充答题思路,明确标注区分 |

---

## 二、技术方案

| 环节 | 选型 | 选型理由 |
|---|---|---|
| 界面 | Gradio | 几行代码即可包成网页聊天界面,能演示、能截图、能生成体验链接 |
| RAG 框架 | LangChain | 主流 RAG 框架,生态成熟 |
| 向量化 | 火山引擎方舟 Embedding API | 云端向量化,512MB 内存约束下避免本地加载模型溢出 |
| 向量库 | FAISS | 轻量、检索快 |
| 生成大模型 | 文心一言(经 5 模型横评选定) | 中文最通俗、结构最清晰、溯源粒度匹配产品设定(详见下方选型说明) |
| 部署 | Render 免费版 | 可行性(能跑 Python)+ 成本(免费)+ 合规(私有仓库不公开面经) |

### RAG 核心链路

```
用户提问
  → 查询理解(指代消解 + 查询改写 + 闲聊过滤,一次调用完成)
  → 向量检索 + 相似度阈值过滤
  → 拼接对话历史 + 相关面经
  → 大模型分层生成答案(真题溯源 / AI 补充思路)
```

## 三、产品设计与关键决策

### 3.1 定制化知识切分

针对面经的问答结构,按"一个 Q+A"作为一个独立知识块切分,而非通用的固定长度切法。从源头保证每个检索单元是完整问答,提升检索质量。

### 3.2 分层回答架构

在"纯面经复读(绝对可信但死板)"与"面经 + AI 补充(实用但需防幻觉)"之间,选择**分层设计**:
- **真题溯源**:严格依据面经,禁止编造来源
- **AI 补充思路**:明确标注为 AI 建议,与真题区分
- **以用户问题为中心**:通过 Prompt 确立"先直接回答用户问题,再关联真题"的原则,防止被检索到的面经角度带偏

### 3.3 多轮对话记忆

多轮追问常带指代和省略(如"那它和运营岗的区别呢"),直接检索会失败。将指代消解、查询改写、闲聊过滤**合并为统一的"查询理解"模块**(一次大模型调用完成),检索用改写句、回答用原问题,并做两层记忆 + 截断防膨胀。

### 3.4 感知性能优化

发现用户点发送后要干等数秒,问题和答案一起蹦出。将"显示问题"和"调用大模型"拆成两步异步执行——问题即时上屏,答案随后生成。真实耗时不变,但用户的心理等待显著缩短。

## 四、检索优化

### 4.1 数据驱动的相似度阈值

阈值不拍脑袋,而是**采样泛问题/精准问题/无关问题的分数分布**后确定(1.1)。采样中发现反直觉现象:短问题因语义信息量少,相似度分数反而偏高、检索不准。

### 4.2 查询改写

针对短问题检索不准的问题,在检索前用大模型将短/模糊问题改写为关键词密集的陈述句。关键设计:**检索用改写句,回答仍用原问题**,避免改写带偏回答。

### 4.3 技术选型匹配阶段

评估过 Rerank 重排序,但判断当前数据规模(约 50 块)下召回候选本就不多、边际收益低,主动列为数据扩大后的储备项——不为炫技而堆技术。

---

## 五、生成模型选型

此前默认使用 DeepSeek,"默认"不等于"最优"。将 5 款主流国产模型(DeepSeek、豆包、通义千问、混元、文心一言)放在**同一批问题、同一套 Prompt、同一温度**下横向对比,按本产品场景定制评估维度:

| 维度 | 为什么重要 |
|---|---|
| 中文表达通俗度 | 用户是紧张备考的求职者,"说人话" > "秀专业" |
| 回答结构清晰度 | 三层结构是否分明、便于快速扫读 |
| 指令遵循 | 能否稳定守住 Prompt 约束 |
| 溯源可信度 | 本产品面经不公开,溯源聚焦到公司维度即可 |

**结论:选定文心一言。** 中文最通俗(同一道题,"结合美团业务场景思考 AI 如何提效" vs "用 LTV/CAC、DAU 渗透率锚定价值")、结构最清晰、溯源粒度恰好匹配"面经不公开"的产品设定。选最适配场景的,而非参数最强的。

## 六、量化评测体系

从"凭手感调优"升级到"用分数说话的数据驱动迭代"。

- **评测集**:20 题按能力维度分四类(精准命中 / 意图错位 / 无关边界 / 多轮追问),每类精准对应项目要防的一个风险。
- **评测方法**:LLM-as-a-Judge,采用**双模型交叉验证**(DeepSeek + 豆包分别打分),三维评分(忠实度 / 答案相关性 / 上下文相关性),避免单一裁判的系统性偏差。
- **发现并修复评测标准自身缺陷**:首次跑分忠实度仅 2.85,反查明细定位到问题不在产品而在评测标准——裁判将已标注的 AI 补充误判为编造。修正忠实度定义后:

| 维度 | 修正前 | 修正后 |
|---|---|---|
| 忠实度 | 2.85 | **3.94** |
| 答案相关性 | 4.23 | 4.54 |
| 上下文相关性 | 3.29 | 4.00 |

## 七、部署:512MB 内存约束下的架构改造

- **问题**:原方案用本地 Embedding 模型(bge-small-zh),模型文件 + PyTorch 运行时超过 Render 免费版 512MB 上限。
- **方案**:将向量化从本地模型改为**云端 Embedding API**(火山引擎方舟),重计算搬到云端,本机只发请求、收向量,内存骤降。
- **排查方法论**:切换 API 后遇到三个连环报错,通过"读官方文档对齐规范 → 用最小 curl 请求隔离变量 → 按真实返回结构改代码"逐一解决。

## 八、如何运行

### 1. 安装依赖

```bash
pip install -r requirements.txt
```

### 2. 配置 API Key(重要)

本项目不把密钥写死在代码里。运行前请设置环境变量:

```bash
export ARK_API_KEY="你的火山引擎 API Key"
```

> 部署到 Render 时,在后台的 **Environment** 里添加 `ARK_API_KEY`。

> 另外需在火山引擎方舟创建一个 **Embedding 接入点**,把其 `ep-` 开头的 ID 填入 `app.py` 的 `向量化接入点ID`(该 ID 不是密码,可留在代码里)。

### 3. 放入面经数据

把整理好的面经 `.txt` 文件(命名如 `公司_岗位_轮次.txt`,内部用 `Q1/A1/Q2/A2...` 结构)放进 `面经数据/` 文件夹。

### 4. 启动

```bash
python app.py
```

浏览器打开 `http://localhost:7860` 即可使用。

## 九、数据说明与合规

- 面经仅**手动收集自用户自愿公开分享的内容**,不使用爬虫批量抓取,规避版权与反爬风险。
- 出于合规考虑,本仓库为**私有仓库**,面经原文不对外公开;评测仅在本地进行,数据不外传。
